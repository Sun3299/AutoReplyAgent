Session 模块（会话管理）
职责：负责对话历史存储、用户状态维护、上下文记忆管理，使用独立数据库与接口，不嵌入任何 AI 相关逻辑。针对新用户/首次咨询场景，可自动调用工具补充基础上下文。
输出：结构化对话历史 ChatHistory

Session 会话层解决方案

1. 串话问题
采用  userId + sessionId  二维维度存储，一个用户可对应多个会话且互不干扰；前端及各渠道请求必须携带  sessionId ，杜绝上下文错乱。
2. 上下文过长导致 Token 超限
仅保留最近 5–10 轮完整对话，较早历史通过 LLM 自动摘要存储，不保存原文；达到长度阈值自动截断，确保不超出大模型限制。
3. 会话过期策略
默认 30 分钟无操作自动过期；支持场景化配置，如咨询类会话 15 分钟、售后类会话 60 分钟；过期后自动归档，不物理删除，便于后续追溯。
4. 多端同步
会话与  userId  绑定，不与设备绑定；同一用户在网页、小程序、微信等任意端登录均可获取同一会话；基于 Redis 实现实时缓存，保证多端状态同步。
5. 存储性能优化
MySQL 用于历史数据持久化，Redis 缓存活跃会话；读取优先走 Redis，写入异步落库，避免拖慢 AI 响应速度。
6. 上下文质量控制
采用三段式存储结构：固定系统提示词、近期完整对话、早期摘要信息，在节省 Token 的同时不丢失关键信息。
7. 刷新/重进上下文丢失
前端将  sessionId  存入 localStorage/cookie，重进时自动携带；后端依据  sessionId  快速重建上下文。
8. 敏感信息合规
会话中不存储手机号、验证码、密码等敏感信息；敏感数据仅存于业务库，会话只保留索引字段；支持一键清空，满足隐私合规要求。
9. 多轮填表状态丢失
在会话中增加  state  状态字段，如 waiting_order_id、waiting_phone 等；用户重新进入后，AI 读取状态并接续提问，无需从头开始。
10. 会话与业务数据一致性
会话仅存储订单号、商品ID等关键字段，不存储业务详情；真实数据实时从业务接口查询，避免数据过期或不一致。

 

Agent 中枢模块（LangChain 核心）
职责：系统核心调度大脑，接收用户问题与对话历史，完成意图判断、RAG 检索决策，并根据不同渠道平台执行后续调度。基于 LangChain Agent 与 ToolCall 实现动作决策，只负责逻辑判断，不执行具体检索、调用或生成操作。
输出：标准化执行指令（RAG 检索/工具调用/直接回复/反问用户）

 

RAG 模块（独立服务）
职责：提供完整知识库能力，包含文档加载、文本分块、向量嵌入、向量库存储与召回检索，可独立开发、独立部署。
对外统一接口：rag.retrieve(query: str) -> list[str]
交互方式：仅通过接口与 Agent 对接，无代码耦合，支持热插拔。

 

Tool 工具集模块（业务能力）
职责：插拔式设计的业务能力集合，每个工具独立实现，不同渠道平台可配置差异化工具集。
核心工具：订单查询工具 OrderTool、物流查询工具 LogisticsTool、退款申请工具 RefundTool、用户信息工具 UserInfoTool。

 

Prompt 模板模块
职责：统一管理系统所有提示词模板，支持按渠道、场景、业务线配置差异化 Prompt，专人维护，修改文案无需改动业务代码。
模板类型：system_prompt、rag_prompt、tool_call_prompt、chat_prompt、reply_prompt。

 

LLM 调用层（大模型封装）
职责：统一封装通义千问、DeepSeek、GPT、豆包等各类大模型 API，屏蔽底层模型差异。
对外统一接口：llm.chat(messages, tools)
优势：切换/新增模型不影响上层业务逻辑，实现无感替换。
